# 机房动力设备智能盘点系统（用户使用与部署手册）

> 本文档只讲两件事：
> 1. 作为使用者，如何使用系统完成盘点工作；
> 2. 作为管理员，如何把系统部署并运行起来。

---

## 一、系统能做什么（给使用者）

本系统用于机房动力设备盘点，核心流程是：

1. 创建盘点任务
2. 上传设备照片
3. 系统自动识别设备信息
4. 人工校对识别结果
5. 确认有效版本
6. 查看汇总并导出结果

系统可管理的信息包括：
- 城市
- 站点
- 设备类型
- 盘点人
- 品牌
- 型号
- 序列号
- 生产日期

---

## 二、使用前准备

### 1）首次使用账号说明
- 首次登录可直接进入系统（免密初始化）
- 建议首次进入后立即设置登录密码
- 设置密码后，后续登录需输入密码

### 2）浏览器建议
推荐使用以下现代浏览器：
- Chrome（推荐）
- Edge

---

## 三、使用流程（重点）

以下是建议的标准操作流程。

### 步骤1：登录系统
1. 打开系统首页
2. 按提示登录（首次可免密）
3. 若系统提示“未配置密码”，请先设置密码

---

### 步骤2：创建盘点任务
在“任务创建”页面填写：
- 城市
- 站点
- 设备类型
- 盘点日期
- 盘点人

点击创建后，系统会自动做判定：
- 若任务完全重复：系统会提示重复
- 若属于更新场景：系统会按更新任务处理

> 建议：任务命名与录入口径保持一致，避免后续统计口径混乱。

---

### 步骤3：上传设备图片
进入任务详情后上传图片，支持一次上传多张。

建议图片要求：
- 清晰、避免严重模糊
- 尽量包含铭牌核心字段（品牌/型号/序列号/生产日期）
- 同一设备建议拍 1~2 张角度

---

### 步骤4：执行识别
可选择两种识别方式：
- 整个任务批量识别
- 单张图片重识别（用于纠错）

识别完成后，系统会生成结构化记录。

---

### 步骤5：人工校对（非常重要）
在识别结果中逐条检查并修正：
- 品牌
- 型号
- 序列号
- 生产日期

建议优先检查：
1. 序列号
2. 型号
3. 品牌
4. 生产日期

> 原因：序列号和型号通常是资产唯一性与统计准确性的关键字段。

---

### 步骤6：确认有效版本
当同一业务维度存在多个任务版本时：
- 在“待确认更新任务”中选择本次应生效的任务
- 执行“确认有效版本”

系统汇总只统计“已确认有效（active）”的数据。

---

### 步骤7：查看汇总与导出
在汇总页面可：
- 查看最新有效统计结果
- 导出 CSV
- 导出 Excel（XLSX）

如需提交给管理方，建议导出 XLSX。

---

## 四、常见使用问题（FAQ）

### Q1：为什么我能看到任务，但汇总里没有？
因为汇总只统计“已确认有效版本（active）”的数据。请检查是否已确认版本。

### Q2：识别错了怎么办？
可在任务详情中手工编辑识别结果，或对单张图片重新识别。

### Q3：候选下拉里有错误值怎么办？
可联系管理员隐藏错误候选项，避免继续污染输入。

### Q4：登录后提示未授权（401）怎么办？
通常是登录状态失效。请重新登录。

## 五、管理员部署指南

> 下面是可直接落地的部署方法。先给最简单的 Windows 一键部署，再给手动部署方式。

## 5.1 部署方式A（推荐）：Windows 一键部署

### 适用场景
- 本地试运行
- 内网演示
- 快速搭建测试环境

### 操作步骤
1. 将项目完整放到本机目录
2. 双击运行根目录下：

```bat
run_all.bat
```

3. 等待脚本自动完成：
   - 安装/检查 uv
   - 安装/检查 Node.js
   - 安装 Python 3.11
   - 创建虚拟环境
   - 安装后端依赖
   - 安装并构建前端
   - 启动前后端服务

4. 启动完成后访问：
   - 前端页面：`http://127.0.0.1:5173`
   - 后端接口文档：`http://127.0.0.1:8000/docs`

### 成功标志
- 能打开前端页面
- 能进入后端 docs
- 能创建任务并上传图片

---

## 5.2 部署方式B：手动部署（可控性更高）

### 环境要求
- Python 3.11
- Node.js 20+
- npm

### 1）后端部署
在项目根目录执行：

```bash
pip install -r requirements.txt
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

后端启动后访问：
- `http://127.0.0.1:8000/docs`

### 2）前端部署（开发模式）

```bash
cd frontend
npm install
npm run dev
```

默认访问：
- `http://127.0.0.1:5173`

### 3）前端部署（生产静态资源）

```bash
cd frontend
npm install
npm run build
```

构建后按你们现有静态站点方案发布 `dist` 目录（如 Nginx/静态服务器）。

## 5.4 运维建议（简版）

### 备份建议
至少备份：
1. `data/db/inventory.db`
2. `uploads/`

建议每日备份一次，保留最近 7~30 天。

### 升级建议
- 升级前先备份数据库与上传目录
- 升级后做一次冒烟测试（登录、创建任务、上传、识别、导出）

### 端口建议
- 后端默认：8000
- 前端默认：5173

如端口冲突，需调整启动参数。

## 七、故障快速处理

### 1）脚本运行失败
请查看命令窗口最后的 `[ERROR]` 行，并按提示处理后重试。

### 2）页面打不开
- 检查前端服务是否启动
- 检查端口是否被占用
- 检查防火墙或安全软件拦截

### 3）接口打不开
- 检查后端服务是否启动
- 访问 `http://127.0.0.1:8000/docs` 验证

### 4）识别效果异常
- 检查图片质量
- 检查模型配置是否正确（若启用了在线模型）
- 先做人工校对再导出

---

## 八、建议的使用规范（提升数据质量）

1. 统一任务录入口径（城市/站点/设备类型命名）
2. 统一拍照规范（优先拍清铭牌）
3. 每次识别后必须人工复核关键字段
4. 每轮盘点结束后必须确认有效版本
5. 导出前先抽检 5% 数据
